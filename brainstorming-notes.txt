My Basic Understanding :

-> By Monthly Quota : We basically need a prepaid balance model for the clients!

The catch:
->> It needs to be custom config, (monthly quota per organization and per feature)
->> Requests for one org can arrive at the same time (concurrent requests)
->> Atomicity needs to be handled because (never negative, never over-served)

Constraints:
->> Scale : 5000 orgs consuming 30 metered features can go upto 50000 orgs consuming the features
->> Traffic: Peak 10k TPS (A few large batches in seconds, then quiet)
->> Deployment : Currently 8 instances, growing (horizontally scaled). 
->> Latency : Less than 10ms per operation (something in-memory RAM based "Redis dependent rate-limiting (for atomicity + speed)")

** Decisioning : 
-> Concurrent correctness : if tokens exhausted or all requests trying to exhaust tokens what needs to be done! 
-> batch correctness (if M < N should we drop all requests ? should we process partially ?)
-> failures & retries (if quota deducted ? What will you do ?  client retries ? double-charge ? are you sure ? )
-> reset & reporting -> Monthly rolling/resetting of credits!! (decision is what monthly means)


Expectations: 
-- Single command execution of the quota-meter with one consumer of quota component running under load

-- Test cases (Proper checks for over-serves, concurrency)
